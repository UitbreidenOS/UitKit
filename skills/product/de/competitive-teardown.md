# Kompetenz Konkurrenz-Teardown

## Wann aktivieren
- Analyse eines spezifischen Konkurrenten vor einer Produktentscheidung
- Identifizierung der Wettbewerbslandschaft Ihrer Kategorie von Grund auf
- Erstellung einer Battle Card für den Vertrieb
- Suche nach unterversorgten Positionierungslücken in einem überfüllten Markt
- Bewertung eines neuen Marktes vor dem Eintritt
- Vorbereitung einer Präsentation für das Vorstand oder Investoren zur Konkurrenz

## Wann nicht verwenden
- Nur Preisforschung — verwenden Sie die Fähigkeit pricing-strategy
- Kunden-Win/Loss-Analyse (benötigt echte Deal-Daten) — dies ist qualitative Wettbewerbsintelligenz
- Rechtliches Konkurrenz-Benchmarking — verwenden Sie die Legal-Fähigkeit
- Echtzeit-Überwachung von Konkurrenzaktivitäten — verwenden Sie die Web-Suche direkt

## Anweisungen

### Vollständiger Konkurrenten-Teardown

```
Führen Sie eine strukturierte Zerlegung von [Konkurrent] durch.

Konkurrent: [Name + URL]
Unser Produkt: [Name + URL]
Warum dieser Konkurrent wichtig ist: [direkter Konkurrent / benachbart / aspirativ]

Teardown-Dimensionen:

1. Positionierung und Messaging:
   - Was behaupten sie auf ihrer Hero-Startseite?
   - Für wen sagen sie, dass es ist (explizites ICP)?
   - In welcher Kategorie positionieren sie sich?
   - Was ist ihr Schlüsseldifferenzierungsmerkmal?
   - Welche Emotion führen sie an (Angst / Aspiration / Vertrauen / Effizienz)?

2. Produkt:
   - Kernfunktionalität (was es tut)
   - Wichtigste Funktionen sichtbar von öffentlichem Produkt + Dokumentation
   - Offensichtlich fehlende Funktionen (was sie nicht tun)
   - Integrations-Ökosystem (womit verbinden sie sich)
   - Mobile App? Desktop-App? Nur API?

3. Preisgestaltung:
   - Modell (pauschal / pro Benutzer / nutzungsbasiert / freemium / nur Unternehmen)
   - Stufen und Preise (wenn öffentlich)
   - Kostenlose Testversion oder freemium verfügbar?
   - Was befindet sich hinter der Paywall?

4. Go-to-Market:
   - Primärer Akquisitionskanal (SEO / bezahlt / PLG / vertriebsgeführt / Gemeinschaft)
   - Content-Strategie (Blog-Themen, YouTube-Präsenz, Podcast)
   - Partnerschaften und Integrationen als Verteilung
   - Verkaufsmodell (Self-Serve / Inside Sales / Field Sales)
   - Zielunternehmensgröße (KMU / Mittelstand / Unternehmen)

5. Stärken und Schwächen:
   - 3-5 Dinge, die sie besser als wir tun
   - 3-5 Dinge, die sie schlecht tun oder gar nicht tun
   - Ihre Positionierungsvulnerabilität (wo sie nicht glaubwürdig konkurrieren können)

6. Kundenstimmung (von Review-Seiten — G2, Capterra, Reddit):
   - Hauptbeschwerden (was Kunden nicht mögen)
   - Hauptlob (was Kunden am meisten lieben)
   - Häufige Wechselgründe (warum Kunden sie verlassen)

Ergebnis: Teardown-Zusammenfassung + 3 strategische Implikationen für unser Produkt oder unsere Positionierung.
```

### Wettbewerbslandschaftskarte

```
Kartographieren Sie die Wettbewerbslandschaft für [Kategorie].

Kategorie: [beschreiben Sie den Problemraum — z.B. „Projektmanagement für Engineering-Teams"]
Unser Produkt: [Name]
Bekannte Konkurrenten: [Liste — oder „identifizieren Sie sie"]

Landschaftskartierung:

Schritt 1 — Alle Konkurrenten identifizieren:
- Direkt: Lösen Sie das gleiche Problem für denselben Kunden
- Indirekt: Lösen Sie das gleiche Problem anders (z.B. Excel als Konkurrenz zu jeder SaaS)
- Benachbart: Lösen Sie ein verwandtes Problem, das mit unserem überlappt
- Aufstrebend: Neue Entrants oder Kategorie-Schöpfer

Schritt 2 — Auf zwei Achsen positionieren (wählen Sie die strategischsten Dimensionen):
   Achse 1: [z.B. Einfach ← → Leistungsstark]
   Achse 2: [z.B. KMU-fokussiert ← → Unternehmens-fokussiert]
   
   Platzieren Sie jeden Konkurrenten auf diesem 2x2.
   Finden Sie: Wo ist das Leerzeichen? Wo ist unsere Positionierung?

Schritt 3 — Kategorie-Dynamiken:
   - Wer ist der Kategorie-Leader (meiste Awareness)?
   - Wer wächst am schnellsten?
   - Wer rückläufig?
   - Konsolidiert sich die Kategorie oder fragmentiert sie?

Schritt 4 — Strategische Implikationen:
   - Können wir ein Segment besitzen, das der Leader ignoriert?
   - Gibt es ein unterversorgtes Persona?
   - Gibt es eine Funktionskombination, die niemand gut zusammen macht?

Produkt: Landschaftskartierungsbeschreibung + 3 Positionierungsmöglichkeiten, die wir übernehmen könnten.
```

### Battle Card

```
Erstellen Sie eine Battle Card gegen [Konkurrent].

Konkurrent: [Name]
Unser häufigster Wettbewerbsszenario: [wo wir gewinnen / wo wir verlieren]
Zielbenutzer: [an wen wir verkaufen, ihre Rolle]
Schlüsselobjekt von diesem Konkurrenten: [„Warum nicht einfach [Konkurrent] verwenden?"]

Battle Card-Struktur:

1. EINZEILEN-DIFFERENZIATOR (damit der Rep es in < 10 Sekunden sagt):
   „[Unser Produkt] tut X, [Konkurrent] nicht, weil [struktureller Grund]."

2. WO WIR GEWINNEN (3 Punkte — konkret, nicht generisch):
   □ [Spezifische Funktion oder Fähigkeit, die sie nicht haben] → Kundenvorteil
   □ [Preis- oder Modellvorteil] → Was der Kunde spart oder vermeidet
   □ [Integrations- oder Workflow-Vorteil] → Spezifischer Workflow, den sie sich Sorgen machen

3. WO SIE GEWINNEN (seien Sie ehrlich — 2 Punkte):
   □ [Sache, die sie besser tun] — „Wenn das ihre oberste Priorität ist, könnten sie recht haben, [Konkurrent] zu wählen"
   □ [Stärke, die wir nicht haben] — Sie anerkennen, dann umleiten

4. WIE MAN ES UMDREHT:
   Für jede Konkurrentenstärke die Umleitung:
   „[Konkurrent] hat [Stärke], aber die Kosten sind [Kompromiss, den sie zahlen]. Unsere Kunden finden, dass [unser Ansatz] besser ist, weil [spezifischer Grund]."

5. BEWEISSTÜCKE:
   - Kundenzitat: „[Genaues Zitat darüber, warum sie uns vor dem Konkurrenten gewählt haben]" — [Rolle, Unternehmen]
   - Gewinngeschichte: [2-Satz-Geschichte eines Deals, den wir gegen sie gewonnen haben]
   - Datenpunkt: [Statistik, die unseren Vorteil zeigt — Geschwindigkeit, NPS, Kundenbindung, was auch immer]

6. MINEN ZU PFLANZEN:
   Fragen, die der Rep stellen sollte, um Konkurrenzschwächen zu offenbaren:
   - „Wie wichtig ist [unsere Stärke] für Sie?"
   - „Haben Sie jemals [Konkurrenzschwäche] erfahren?"
   - „Was passiert, wenn Sie [Sache, die sie nicht tun] brauchen?"

Schreiben Sie die Battle Card für meinen spezifischen Konkurrenten.
```

### Win/Loss-Analyse

```
Entwerfen Sie einen Win/Loss-Analyseprozess für [Unternehmen].

Aktueller Stand: [kein Win/Loss-Tracking / ad hoc / systematisch]
Verkaufszyklus: [Self-Serve / Inside Sales / Unternehmen]
Deal-Volumen: [X Deals/Monat]

Win/Loss-Rahmen:

1. Wen interviewer:
   - Gewonnene Deals: 30-60 Tage nach Abschluss interviewen (Flitterwochen sind vorbei, echte Gründe kommen raus)
   - Verlorene Deals: innerhalb von 2 Wochen interviewen (Erinnerung verblasst, Konkurrenten bewegen sich)
   - Interviewer den tatsächlichen Käufer, nicht den Champion — die Begründung des wirtschaftlichen Käufers unterscheidet sich

2. Fragen zum Gewinn-Interview:
   - „Was hat Sie dazu bewogen, uns gegenüber den Alternativen zu wählen?"
   - „Wer hätte Sie fast dazu gebracht, jemand anderen zu wählen?"
   - „Was hätte Sie dazu gebracht, uns NICHT zu wählen?"
   - „Welche Funktion oder welcher Moment hat bestätigt, dass wir recht hatten?"
   - „Was sagen Sie Kollegen, wenn Sie uns empfehlen?"

3. Fragen zum Loss-Interview:
   - „Was hat Sie dazu gebracht, [Konkurrenten / nichts] stattdessen zu wählen?"
   - „Was haben wir im Prozess gut gemacht?"
   - „War es Produkt, Preis, Prozess oder Timing?"
   - „Was hätte anders sein müssen, um uns zu wählen?"
   - „Ist die Tür offen in der Zukunft? Was müsste sich ändern?"

4. Analysehäufigkeit:
   - Monatlich: Win/Loss-Rate nach Konkurrenten, nach Segment, nach Deal-Größe
   - Vierteljährlich: Themensynthese → Produktlücken, Positionierungsanpassungen
   - Jährlich: vollständige Landschaftsüberprüfung

5. Ausgabeformat:
   - Gewinnungsgründe (Top 3) + % Häufigkeit
   - Verlustgründe (Top 3) + % Häufigkeit
   - Konkurrent-spezifische Muster
   - Produktlücken-Hitzekarte (was Kunden wollten, das wir nicht hatten)

Richten Sie diesen Prozess für meine Sales-Motion und Team-Größe ein.
```

## Beispiel

**Benutzer:** Wir erstellen interne Developer Tools. Notion hat gerade Developer Docs gestartet. Sollten wir uns Sorgen machen? Führen Sie einen Teardown durch.

**Claudees strategisches Framing:**

Notions Eintritt in Ihren Raum ist eine Bedrohung für die untere Sparte (kleine Teams, niedriges Budget), schafft aber eine Gelegenheit im Premium-Segment (wo Notion immer „gut genug" statt maßgeschneidert sein wird).

**Notions strukturelle Beschränkungen (Dinge, die sie nicht ändern können):**
1. Horizontales Produkt — kann nicht tief in entwicklerspezifische Funktionen gehen (Code-Review, PR-Integration, Deployment-Dokumentation), ohne ihre Nicht-Dev-Benutzer zu entfremden
2. Block-basierter Editor — großartig für strukturierte Dokumente, schlecht für API-Referenz oder auto-generierte Dokumente aus Code
3. Kein natives Code-Indexing — sie können nicht aus Ihrer Codebasis extrahieren und Dokumente automatisch synchron halten
4. Enterprise Developer Tooling erfordert SSO/SCIM, Audit-Logs, Git-Integration — Notions Unternehmensbereich ist für HR und Ops ausgelegt, nicht Platform Teams

**Wo Sie sofort differenzieren sollten:**
- Auto-Sync mit Code (Dokumentation, die sich aktualisiert, wenn sich Code ändert — Notion kann das nicht, ohne ihr Modell zu brechen)
- Developer-native Formate: OpenAPI-Rendering, Code-Spielplatz, Changelog von Git-Commits
- IDE-Integration (VS Code, JetBrains — Notion hat hier keinen Spielraum)

**Ihre Messaging-Antwort:**
Erwähnen Sie Notion nicht beim Namen. Stattdessen:
Hero: „Dokumentation, die mit Ihrem Code synchron bleibt — automatisch."
Dies ist das, was Notion strukturell nicht anbieten kann, als Vorteil (nicht als Vergleich) formuliert.

**Fazit:** Machen Sie sich Sorgen um Notion, das das SMB-Segment nimmt, das Sie akquirieren wollten. Machen Sie sich keine Sorgen um Notion, das Ihre bestehenden Enterprise- oder Mid-Market-Kunden nimmt, die Tiefe brauchen.
