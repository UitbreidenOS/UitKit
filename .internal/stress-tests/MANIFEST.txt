PRIMARY EXECUTABLE
-------------------

swarm-sandbox-stress.js (961 lines, 32 KB)

Classes:
  - StressSandbox(name, agentCount, stressConfig)
    * initialize(): Create sandbox directories and agent configs
    * execute(): Simulate workload with memory allocation
    * cleanup(): Recursive directory deletion with timeout
    * getStats(): Return lifecycle metrics

  - StressTestScenario(name, stressLevel, stressConfig)
    * recordMetric(name, value): Track measurement data
    * recordError(error, context): Log failures
    * getStats(): Calculate percentiles and rates

Stress Test Functions:
  1. testInitializationStorm(stressLevel, config)
     - Rapid concurrent sandbox creation
     - Tests init bottlenecks and throughput
     - Measures creation time distribution

  2. testExecutionPressure(stressLevel, config)
     - Sustained workload with memory stress
     - Tests degradation under load
     - Detects cascading failures

  3. testCleanupUnderPressure(stressLevel, config)
     - Aggressive cleanup with tight timeouts
     - Tests resource recovery
     - Measures cleanup timeout rate

  4. testCascadingFailure(stressLevel, config)
     - Sequential creation with failure injection
     - Stops when failure rate exceeds 30%
     - Tests failure propagation

Report Generation:
  - generateReport(scenarios): Aggregate metrics
  - saveReport(report, format): Write JSON/Markdown

Utilities:
  - generateId(): UUID generation
  - log(msg, level): Colored logging
  - getMemoryUsage(): Process memory stats
  - shouldInjectFailure(rate): Random failure injection
  - allocateMemory(sizeKb): Simulate memory pressure

Configuration:
  STRESS_LEVELS: {low, medium, high, extreme}
    - Each level defines sandbox count, agent count, memory spikes,
      failure rates, and cleanup timeouts

  STRESS_ROOT: ~/.claude/stress-test-sandboxes
    - Temporary sandbox directories
    - Report output directory

================================================================================
TEST SCENARIOS SUMMARY
================================================================================

1. INITIALIZATION STORM
   Purpose: Measure rapid sandbox creation throughput
   Load: 50-200 concurrent init operations
   Metrics: Init time, memory peak, failure count
   Success Criterion: p99 < 1000ms, failure rate < 15%

2. EXECUTION PRESSURE
   Purpose: Test system stability under sustained workload
   Load: Concurrent execution with memory allocation
   Metrics: Exec time, cascading failures, memory degradation
   Success Criterion: p99 < 8000ms, cascading failures < 10

3. CLEANUP UNDER PRESSURE
   Purpose: Test resource recovery and timeout resilience
   Load: Aggressive cleanup batches with tight timeouts
   Metrics: Cleanup time, timeout rate, memory recovery
   Success Criterion: Timeout rate < 5%, memory recovery > 70%

4. CASCADING FAILURE
   Purpose: Detect failure propagation patterns
   Load: Sequential creation with failure injection
   Metrics: Sandboxes created before cascade, failure trajectory
   Success Criterion: Creates > 60% of target, gradual failure growth

================================================================================
METRICS AND FORMULAS
================================================================================

Timing Metrics (ms):
  - initializationTimes[]: Per-sandbox init duration
  - executionTimes[]: Per-sandbox execution duration
  - cleanupTimes[]: Per-sandbox cleanup duration
  Computed: min, max, avg, p95, p99

Memory Metrics (MB):
  - memoryStart: Heap used at scenario start
  - memoryPeak: Maximum heap during scenario
  - memoryEnd: Heap used at completion

Count Metrics:
  - successCount: Sandboxes completed successfully
  - failureCount: Sandboxes that failed
  - timeoutCount: Cleanup operations that timed out
  - cascadingFailures[]: Failed due to upstream failures

Rate Metrics:
  - failureRate = (failureCount / totalSandboxes) × 100
  - successRate = (successCount / totalSandboxes) × 100
  - timeoutRate = (timeoutCount / totalSandboxes) × 100

Percentile Calculation:
  For sorted array of N observations:
  p95_index = ceil(N × 0.95) - 1
  p99_index = ceil(N × 0.99) - 1
  p95 = array[p95_index]
  p99 = array[p99_index]

================================================================================
INTEGRATION WITH LOAD TESTS
================================================================================

Complementary Testing Strategy:

Load Tests (swarm-sandbox-load.js):
  - Focus: Capacity, throughput, normal operation
  - Failure Rate: 0-5% (natural failures)
  - Cleanup: Standard batching
  - Duration: Steady state

Stress Tests (swarm-sandbox-stress.js):
  - Focus: Failure modes, resource exhaustion, recovery
  - Failure Rate: 5-25% (injected failures)
  - Cleanup: Aggressive timeouts
  - Duration: Peak conditions

Together They Measure:
  1. Load tests: What's the max throughput?
  2. Stress tests: What happens at breaking point?
  3. Combined: What's the safe operating envelope?

TROUBLESHOOTING GUIDE
================================================================================

Problem: High Cleanup Timeouts
  Symptom: timeoutCount > 5% of sandboxes
  Cause: Cleanup batch size too large, filesystem I/O bottleneck
  Fix: Reduce cleanupBatchSize in stress config
       Increase cleanupTimeoutMs
       Check disk I/O usage

Problem: Cascading Failures
  Symptom: cascadingFailures[] array populated
  Cause: Initialization failure rate too high, resource contention
  Fix: Reduce failureInjectionRate
       Implement file descriptor recycling
       Check ulimit: ulimit -n

Problem: Memory Not Recovering
  Symptom: memoryEnd > 80% of memoryPeak
  Cause: Buffers not freed, garbage collection not triggered
  Fix: Verify buffer cleanup in StressSandbox.cleanup()
       Add global.gc() call before cleanup
       Check for file handle leaks

Problem: Slow Initialization
  Symptom: initTime.p99 > 1000ms
  Cause: Filesystem I/O bottleneck, disk contention
  Fix: Monitor disk I/O during test
       Reduce concurrentSandboxes
       Use faster storage

Problem: Out of Memory
  Symptom: Process crashes during memory allocation
  Cause: Insufficient available memory for stress level
  Fix: Close other applications
       Reduce stress level (low/medium)
       Increase system memory
